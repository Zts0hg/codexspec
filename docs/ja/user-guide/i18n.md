# 国際化

CodexSpec は **LLM 動的翻訳** によって複数の言語をサポートし、GitHub Pages 向けに **多言語ドキュメント** を提供しています。

## コマンドテンプレート翻訳

### 仕組み

1. **単一の英語テンプレート**: すべてのコマンドテンプレートは英語のまま
2. **言語設定**: プロジェクトは優先出力言語を指定
3. **動的翻訳**: Claude が実行時にコンテンツを翻訳

### 言語の次元

CodexSpec は言語を独立に設定可能な 4 つの次元に分割しています。`output` がベースであり、他はこれを上書きし、未設定時はこれに (次いで `en`) フォールバックします。

| 次元 | `config.yml` キー | init で設定 | 後から設定 | 制御対象 | フォールバック先 |
|-----------|------------------|-------------|-----------|----------|---------------|
| Output (ベース) | `output` | `--lang` | `config --set-lang` | 他 3 つのベース | `en` |
| Interaction | `interaction` | `--interaction-lang` | `config --set-interaction-lang` | LLM との対話 + CLI 出力 | output → `en` |
| Document | `document` | `--document-lang` | `config --set-document-lang` | 生成される spec/plan/tasks | output → `en` |
| Commit | `commit` | `--commit-lang` | `config --set-commit-lang` | git コミットメッセージ | output → `en` |
| Templates | `templates` | — | — | コマンドテンプレートのソース (常に `en`) | — |

### 言語の設定

#### 初期化時

```bash
# 中国語で出力 (output ベースを設定)
codexspec init my-project --lang zh-CN

# 完全に非対話型: zh-CN ベース、英語のコミットメッセージ
codexspec init my-project --lang zh-CN --commit-lang en

# 各次域を明示的に設定 (スクリプト可能、プロンプトなし)
codexspec init my-project \
  --interaction-lang zh-CN --document-lang en --commit-lang en
```

TTY 環境で `--lang` を付けずに (かつ 3 つの次元フラグすべてを付けずに) 初回 init を実行するとベース言語の入力を求められます。非 TTY (CI/スクリプト) では `en` がデフォルトです。`init` を再実行しても、指定しなかった言語キーは保持されます。

#### 初期化後

```bash
# 現在の設定を表示
codexspec config

# 単一次元を変更
codexspec config --set-lang zh-CN
codexspec config --set-interaction-lang zh-CN
codexspec config --set-document-lang en
codexspec config --set-commit-lang en

# サポートされている言語を一覧表示
codexspec config --list-langs
```

### 設定ファイル

`.codexspec/config.yml`:

```yaml
version: "1.0"

language:
  output: "zh-CN"        # ベース言語。以下 3 つはこれに (次いで "en") フォールバック
  interaction: "zh-CN"   # LLM との対話 + codexspec CLI 出力 (省略可 → デフォルトは output)
  document: "en"         # 生成される requirements/spec/plan/tasks (省略可 → デフォルトは output)
  commit: "en"           # git コミットメッセージ (省略可 → デフォルトは output)
  templates: "en"        # "en" のまま

project:
  ai: "claude"
  created: "2025-02-15"
```

## GitHub Pages ドキュメントの i18n

CodexSpec のドキュメントサイトは MkDocs と mkdocs-i18n プラグインを使用して 8 つの言語をサポートしています。

### サポートされている言語 {#supported-languages}

| コード | 言語 |
|------|----------|
| `en` | English (デフォルト) |
| `zh` | 中文简体 |
| `ja` | 日本語 |
| `ko` | 한국어 |
| `es` | Español |
| `fr` | Français |
| `de` | Deutsch |
| `pt-BR` | Português (Brasil) |

### ディレクトリ構造

```
docs/
├── en/                 # English (ソース)
│   ├── index.md
│   ├── user-guide/
│   └── ...
├── zh/                 # 中国語翻訳
├── ja/                 # 日本語翻訳
├── ko/                 # 韓国語翻訳
├── es/                 # スペイン語翻訳
├── fr/                 # フランス語翻訳
├── de/                 # ドイツ語翻訳
└── pt-BR/              # ポルトガル語 (ブラジル) 翻訳
```

### ドキュメントの翻訳 (メンテナー専用)

> **注意**: `/codexspec:translate-docs` および `/codexspec:check-i18n-semantics` コマンドは **CodexSpec メンテナー専用ツール** です。これらは本リポジトリの MkDocs i18n レイアウト (`docs/{lang}/`) と、リポジトリ内にのみ存在する用語集 `docs/i18n/glossary.yml` に密結合しています。`codexspec init` によってユーザープロジェクトに配布されることは **なく**、エンドユーザー向けのスラッシュコマンドとしても提供され **ません**。
>
> CodexSpec 本体を開発している場合、これらのコマンドは `.claude/commands/codexspec/` に配置されており、他のスラッシュコマンドと同じ方法で呼び出せます。自身のプロジェクトのドキュメントを翻訳したいエンドユーザーは、Claude を直接利用し、好みのワークフローに組み込んでください。

CodexSpec メンテナーは `/codexspec:translate-docs` コマンドを使用してドキュメントを翻訳します:

```bash
# 中国語に翻訳
/codexspec:translate-docs --lang zh

# 特定のファイルを日本語に翻訳
/codexspec:translate-docs user-guide/installation.md --lang ja

# 差分翻訳 (変更されたファイルのみ)
/codexspec:translate-docs --lang ko --incremental

# プレビューモード (ファイル変更なし)
/codexspec:translate-docs --lang es --dry-run
```

### 翻訳用語集

`docs/i18n/glossary.yml` の用語集は一貫した用語使用を保証します:

```yaml
version: "1.0"

# 英語のままにする用語
keep_english:
  - CLI
  - API
  - YAML
  - JSON

# 一般的な用語の翻訳
translations:
  zh:
    specification: "规格说明"
    constitution: "宪法"
    task: "任务"

# 特殊なケースのスマートルール
rules:
  - pattern: '\b(CLI|API)\b'
    action: keep
```

### 翻訳方法論

翻訳は、英語を逐語的に反映するのではなく、あなたの言葉で自然に読めることを目指します：

- **意味を優先。** 各段落は伝えたい内容を捉え、対象言語の表現と文構造で改めて表現します。
- **英語のままが分かりやすい場合は英語を維持。** 適切な母語の語がない、あるいは英語のほうが読者に伝わりやすい技術用語は、無理に訳さず英語のまま残します。例えば "AI coding agent" は、生硬な直訳ではなく「AI コーディングアシスタント」のような自然な表現になります。
- **母語としての自然さ。** 最初からその言語で書かれたかのように読める文章を目指します。

### 品質チェック

#### 構造チェック

すべての言語ディレクトリが同じファイル構造を持っていることを確認します:

```bash
./scripts/bash/check-i18n-structure.sh
```

#### 完全性チェック

翻訳内の未翻訳の英語コンテンツを検出します:

```bash
./scripts/bash/check-i18n-completeness.sh
```

#### 意味チェック (メンテナー専用)

AI 駆動の意味的一貫性分析は、内部コマンドを介して CodexSpec メンテナーのみが利用できます (上記のメンテナー専用に関する注意を参照してください):

```bash
/codexspec:check-i18n-semantics --lang zh
```

### CI/CD 統合

`.github/workflows/docs-i18n.yml` ワークフローは **ビルドゲート専用** です。`docs/**` / `mkdocs.yml` の変更があるたびに (`main` への push とプルリクエストで) すべての言語バージョンに対して `mkdocs build --strict` を実行し、ビルドエラーやリンク切れ、不正なコンテンツを検出します。翻訳は一切行いません。

翻訳は `/codexspec:translate-docs` (前述) で手動に生成・コミットされ、英語ソースとすべての翻訳がレビューされた 1 つのアトミックなコミットにまとめられます。CI での自動翻訳は意図的に削除されました。以前のジョブは実際には一度も実行されず (不正な `if` によりスキップされていた)、常駐の API キーを必要とし、レビューされていない翻訳を `main` に直接コミットするものでした。

## メリット

- **ゼロ翻訳メンテナンス**: 複数のテンプレートバージョンを維持する必要がない
- **常に最新**: テンプレートの更新はすべての言語に反映
- **コンテキスト認識**: 技術用語は適切な場合に英語のまま
- **自動品質管理**: CI が翻訳の一貫性を保証
- **AI アシスト**: Claude がコンテキスト認識の翻訳を提供
