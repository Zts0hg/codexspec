## Needs

### NEED-001: Agent-brand-agnostic structured-question tool invocation in templates

- **Status**: confirmed
- **Statement**: The 2 explicit `AskUserQuestion` tool-name references in codexspec command templates must be rewritten to an agent-brand-agnostic form that correctly triggers structured questioning under both Claude Code (`AskUserQuestion`) and Codex (`request_user_input`), degrading naturally to plain-text questioning when the host tool is unavailable.
- **Locations**: `templates/commands/constitution.md:56`, `templates/commands/config.md:70`
- **Rationale**: codexspec now supports Codex (`project.ai: "both"`); a template that hardcodes `AskUserQuestion` names a tool Codex does not possess, so the instruction silently fails under Codex.
- **User Evidence**: "当前codexspec提供的命令中会要求使用ask user question，但这是claude code中的工具，我们现在已经兼容codex，所以需要检查是否在codex中也可以调用到ask user question工具，或者是codex有其他工具名需要兼容。"
- **Confirmed At**: 2026-07-11

## Constraints

### CON-001: Codex `request_user_input` differs from Claude Code `AskUserQuestion`

- **Status**: confirmed
- **Statement**: Codex's `request_user_input` differs from Claude Code's `AskUserQuestion` in tool name and schema: different name; Codex requires an `id` (snake_case) per question; 2–3 options (vs 2–4); 1–3 questions, prefer 1 (vs 1–4); Codex has no `multiSelect` but has `autoResolutionMs`; available only on the root thread and only in Plan mode (or Default mode behind the `DefaultModeRequestUserInput` feature flag). The client auto-adds an "Other" option in both.
- **User Evidence**: codex-rs source — `codex-rs/core/src/tools/handlers/request_user_input_spec.rs:8` (`REQUEST_USER_INPUT_TOOL_NAME = "request_user_input"`), `request_user_input.rs`, `codex-rs/tools/src/tool_config.rs:38`, `codex-rs/protocol/src/config_types.rs:648` (`allows_request_user_input` matches only `ModeKind::Plan`).

## Decisions

### DEC-001: Scope limited to the 2 explicit tool-name references

- **Status**: confirmed
- **Decision**: Only `constitution.md:56` and `config.md:70` are changed. Prose "ask the user" instructions in other templates are left unchanged because they do not name a tool and are already agent-brand-agnostic.
- **Alternatives Rejected**: Auditing and rewriting all "ask the user" prose across every template (rejected as out of scope — those instructions already work under both agents).
- **Reason**: User chose the minimal scope.
- **User Evidence**: User selected "仅 2 处显式引用" (only the 2 explicit references).

### DEC-002: Name the tools, but do not name agent brands

- **Status**: confirmed
- **Decision**: Rewrite mechanism names the two known tool names (`AskUserQuestion` / `request_user_input`) as a soft, non-exhaustive list using "e.g., `AskUserQuestion` or `request_user_input`", but does NOT name agent brands ("Claude Code" / "Codex"). Each agent calls whichever tool it possesses. This gives the agent a concrete call target while staying forward-compatible with future agents (Gemini, Copilot, etc.).
- **Exact wording — `constitution.md:56`**: `**IMPORTANT**: Use the host agent's structured-question tool (e.g., \`AskUserQuestion\` or \`request_user_input\`) to present structured choices.`
- **Exact wording — `config.md:70`**: `If configuration exists and no \`--view\` flag, present the management menu using the host agent's structured-question tool (e.g., \`AskUserQuestion\` or \`request_user_input\`):`
- **Alternatives Rejected**:
  - (A) Name both tools WITH agent-brand labels ("Claude Code: `AskUserQuestion`; Codex: `request_user_input`") — rejected: hardcodes agent brands, hurts future-agent compatibility.
  - (B) Pure neutral wording naming no tool ("use the host agent's native question/choice tool") — rejected: risks the agent misreading it as a prose question and losing the structured-choice intent.
  - (C) Name both tools + explicit fallback clause — rejected: heavier than needed; under this mechanism the agent degrades naturally because no single tool is forced.
- **Reason**: User proposed naming the tool while dropping agent-brand attribution; confirmed the "e.g. soft list" phrasing.
- **User Evidence**: User selected "e.g. 软列举 (推荐)".

### OPEN-003: JSON example schema mismatch under Codex

- **Status**: open
- **Statement**: Each of the 2 rewrite sites is immediately followed by a JSON example in Claude Code `AskUserQuestion` schema (fields `question`/`header`/`options`, no `id`). Under DEC-002 a Codex agent is explicitly pointed at `request_user_input` and may copy the example schema, failing validation because of the missing `id` (and Codex's 2–3 option limit).
- **Proposed default**: Keep the example as a structural illustration, but add a one-line note that the field schema depends on the host tool (Codex `request_user_input` requires an `id` per question and accepts 2–3 options), and the agent must follow its host tool's actual schema.
- **Why It Matters**: Without the note, naming `request_user_input` could actively mislead the Codex agent into an invalid call.
- **Owner**: User
