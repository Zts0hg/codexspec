## Context

The SDD pipeline runs as a sequence of slash commands (`specify → generate-spec → spec-to-plan → plan-to-tasks → implement-tasks`), each manually triggered today. Three generation commands already end with an Automatic Review Loop yielding an Overall Status (`PASS` / `PASS_WITH_WARNINGS` / `NEEDS_REVISION` / `BLOCKED`), and `plan-to-tasks` already conditionally auto-invokes `/codexspec:analyze` on a passing verdict (commit #17). This feature generalizes that proven pattern into an opt-in `workflow.auto_next` mode that advances the chain automatically once a stage has passed.

**Verified repository facts governing the design:**

- Config keys such as `git.main_branches` are read by the **agent** from `.codexspec/config.yml` at template-execution time; Python (`i18n.py`) only reads `language.*` and performs **no schema validation** that would reject an unknown `workflow` section. → The feature is **template-only**; no Python change is required, and a user adding `workflow.auto_next` cannot break config parsing.
- The default config generated by `init` (`i18n.py:generate_config_content`) emits only `language`/`project`; `git.*` is not in it either. → `workflow.*` follows the same opt-in convention; no default-config change.
- `/codexspec:config` manages only language settings (it does not manage `git.*`). → Leaving it unchanged is consistent.
- Distributed command templates live in `templates/commands/` (source of truth); `.claude/commands/codexspec/` is a self-bootstrap install artifact that must not be hand-edited (constitution). → Edits go to `templates/commands/`, then the derived copies are re-synced.
- `tests/test_sdd_workflow_templates.py` asserts command-template content via `read_command(name)`; `test_en_translation_catalog_matches_template_frontmatter` forces an en.json sync whenever **frontmatter** changes. → The auto-next change is **body-only**; frontmatter must stay untouched so translation catalogs are unaffected.

## Goals / Non-Goals

**Goals:**

- Add an opt-in `workflow.auto_next` mode that auto-invokes the next chain command once the current stage has passed.
- Reuse each command's existing review verdict (and `specify`'s Completion) as the gate.
- Preserve current manual behavior by default.

**Non-Goals:**

- No Python code, config schema, default-config, or `/codexspec:config` wizard changes.
- No per-transition opt-out (OUT-002); no confirmation prompt before `implement-tasks` (DEC-001); no auto-invoking optional helpers (OUT-001).

## Tech Stack

- **Language**: Markdown (command templates) + Python (tests only).
- **No new dependencies.** No build/packaging change (`templates/` is already shipped).

## Architecture Overview

The auto-next behavior is a conditional section appended to four distributed command templates. Each section instructs the agent to: read `workflow.auto_next`, evaluate the stage-passed precondition, emit a one-line notice in the interaction language, and invoke the next slash command exactly once. The pattern is identical in spirit and placement to the existing "Automatic Cross-Artifact Analysis" block in `plan-to-tasks.md`.

**Covers**: REQ-001, REQ-002, REQ-003, REQ-004, REQ-005, REQ-006, REQ-008, REQ-009, REQ-010, NFR-002, NFR-003, NFR-004

```
specify ──(Completion confirmed)──► generate-spec ──(review PASS/PASS_WITH_WARNINGS)──► spec-to-plan
                                                                                              │
                                                   review PASS/PASS_WITH_WARNINGS ◄──────────┘
                                                                                              │
                                                                                              ▼
                                                                                       plan-to-tasks
                                                                                              │
                                                   review PASS/PASS_WITH_WARNINGS ◄──────────┘
                                                                                              │
                                                                                              ▼
                                                                            analyze (existing, read-only, non-gating)
                                                                                              │
                                                                                              ▼
                                                                                       implement-tasks  (TERMINAL — no auto-next section)

Each "─►" edge fires ONLY when workflow.auto_next == true; notice line emitted before each edge.
```

### Component C1 — `specify.md` Auto-Next section  · Covers: REQ-003, REQ-005, REQ-006, REQ-010

Add an "## Auto-Next Chain Advance" section after the Completion section. It instructs the agent to read `workflow.auto_next` (default `false`; only literal `true` enables — Assumption A1) and, when enabled AND `specify` has reached Completion (all discovery criteria met and the user confirmed the **final** stage summary — not each intermediate topic confirmation), to emit one notice line and invoke `/codexspec:generate-spec <feature-dir>` exactly once. Suggested notice: `auto_next: requirements confirmed → invoking /codexspec:generate-spec <feature-dir>`.

### Component C2 — `generate-spec.md` Auto-Next section  · Covers: REQ-002, REQ-004, REQ-005, REQ-006, REQ-010

Add the section after the Automatic Review Loop. Gate: `workflow.auto_next: true` AND the review loop's Overall Status is `PASS` or `PASS_WITH_WARNINGS`. Action: notice + invoke `/codexspec:spec-to-plan <feature-dir>`. Explicit stop conditions: do not auto-invoke when disabled, or when the review stopped at `NEEDS_REVISION` / `BLOCKED` or stopped early (mirroring the analyze precedent's early-stop handling). Suggested notice: `auto_next: review passed → invoking /codexspec:spec-to-plan <feature-dir>`.

### Component C3 — `spec-to-plan.md` Auto-Next section  · Covers: REQ-002, REQ-004, REQ-005, REQ-006, REQ-010

Same shape as C2, advancing to `/codexspec:plan-to-tasks <feature-dir>`.

### Component C4 — `plan-to-tasks.md` Auto-Next section  · Covers: REQ-002, REQ-004, REQ-005, REQ-006, REQ-008, REQ-009, REQ-010

Add the section **after** the existing "Automatic Cross-Artifact Analysis" (analyze) section, before the Output Summary. Gate: `workflow.auto_next: true` AND the review loop passed (analyze has already run). Action: notice + invoke `/codexspec:implement-tasks <feature-dir>` **with no confirmation prompt** (DEC-001 / REQ-009). State explicitly that analyze's findings do not block this advance (CON-005 / REQ-008). Same stop conditions as C2.

### Component C5 — Sync derived self-bootstrap copies  · Covers: REQ-010 (source-of-truth), constitution compliance

After editing the four templates, re-sync `.claude/commands/codexspec/{specify,generate-spec,spec-to-plan,plan-to-tasks}.md` from `templates/commands/` (e.g., re-run `codexspec init` or copy the four files). This makes the change live in this repo for self-testing without violating the "never hand-edit `.claude/`" rule.

### Component C6 — Contract tests  · Covers: REQ-001..010, NFR-002, NFR-004

Extend `tests/test_sdd_workflow_templates.py` with a parametrized test asserting, for each of `specify`, `generate-spec`, `spec-to-plan`, `plan-to-tasks`: the template contains an auto-next section referencing `workflow.auto_next`, the correct next command, the notice, and the stop conditions (`NEEDS_REVISION`/`BLOCKED`). Add targeted asserts: `specify`'s section references Completion (not intermediate confirmation); `plan-to-tasks` references `implement-tasks` and that analyze is non-blocking. Add a guard that `implement-tasks.md` contains **no** auto-next advance (terminal — REQ-007).

### Component C7 — Compatibility test  · Covers: NFR-001

Add a test that a `config.yml` containing `workflow: { auto_next: true }` is parsed by `yaml.safe_load` and that language resolution (`i18n`) remains correct — proving an unknown `workflow` key does not break parsing or existing behavior (backwards compatibility).

## Decisions

### Decision 1: Template-only implementation (no Python)

**Context**: The auto-next behavior must evaluate a config flag and invoke the next command.

**Options Considered**:

1. Template-only: instruct the agent to read `workflow.auto_next` and conditionally invoke the next command (mirrors `git.main_branches` and the analyze precedent).
2. Add Python config parsing + a runtime invocation hook.

**Decision**: Option 1.

**Rationale**: Config keys are already read by the agent at template-execution time (`git.main_branches`), and the analyze auto-invoke is an existing, proven template-only mechanism. Option 2 has no precedent and would require a new invocation mechanism — over-engineering. Smallest architecture satisfying the requirements.

**Covers**: REQ-010, REQ-001

**Decision Level**: Plan-level technical decision; does not change confirmed product scope.

### Decision 2: Do not extend `/codexspec:config` or the default config

**Context**: Where should `workflow.auto_next` be documented/managed?

**Decision**: Leave the config wizard and default config unchanged; `workflow.*` is opt-in and user-managed, exactly like `git.*`.

**Rationale**: Consistency with the existing `git.*` convention; smallest architecture; the spec requires only that the option be readable, which the templates provide.

**Trade-off**: Discoverability is via docs only (addressed by the optional docs phase).

**Covers**: CON-004, NFR-002, NFR-003

### Decision 3: Body-only edits; do not touch frontmatter

**Context**: `test_en_translation_catalog_matches_template_frontmatter` forces an en.json + localized-JSON sync on any frontmatter change.

**Decision**: All four edits are body-only (new sections); frontmatter stays byte-identical.

**Rationale**: Avoids cascading translation-catalog work and keeps the change isolated. Verified repository fact constraining the implementation.

**Covers**: (implementation constraint; supports REQ-010 delivery without scope creep)

### Decision 4: Section placement and shared contract

**Decision**: Each section is named "Auto-Next Chain Advance", placed after the relevant review/Completion/analyze step and before the Output Summary, and shares one contract: read `workflow.auto_next` (only `true` enables) → check stage-passed → stop on disabled/`NEEDS_REVISION`/`BLOCKED`/early → emit one notice (interaction language) → invoke next command exactly once.

**Rationale**: Mirrors the analyze precedent's placement and determinism; the advance must occur after the verdict is known.

**Covers**: REQ-002, REQ-003, REQ-004, REQ-005, REQ-010

## Risks / Trade-offs

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LLM mis-evaluates the gate or invokes more than once | Medium | Medium | Deterministic, explicit template language; mirror the proven analyze block; contract tests (C6) assert the section + gating strings. |
| Self-bootstrap drift (templates edited, `.claude/` not re-synced) | Medium | Low (this repo's live behavior only) | C5 sync step + a test asserting the four `.claude/` copies match `templates/`. |
| Unattended `implement-tasks` writes/commits code (DEC-001) | Low | High | Confirmed trade-off; per-stage review gates must pass first; documented for awareness (no spec change). |
| Non-boolean value silently treated as enabled | Low | Medium | Templates state "only literal `true` enables" (Assumption A1); C7 covers parsing. |

**Accepted trade-off**: No per-transition opt-out (OUT-002) and no pre-implement confirmation (DEC-001).
