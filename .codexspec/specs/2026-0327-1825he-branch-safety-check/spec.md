# 功能：Git 分支安全检测

## 概述

在 CodexSpec 的 slash 命令中添加自动 Git 分支检测功能，当用户在主分支（main/master）上执行功能开发相关命令时，系统会自动提示用户是否需要创建新功能分支，防止直接在主分支上进行开发工作。

## 目标

- 防止用户意外在主分支上直接开发新功能
- 提供友好的交互式分支创建体验
- 与现有 `create-new-feature.sh` 脚本集成
- 保持用户现有工作流习惯

## 用户故事

### 故事 1：主分支上的新功能开发

**作为** 开发者
**我想要** 在执行 `/codexspec:specify` 时自动检测当前分支
**以便** 避免意外在主分支上直接开发

**验收标准：**

- [ ] 当我在 main/master 分支执行命令时，系统提示我创建新分支
- [ ] 我可以选择创建新分支或继续在当前分支工作
- [ ] 如果选择创建分支，系统自动完成分支创建和切换

### 故事 2：功能分支上的继续开发

**作为** 开发者
**我想要** 在已创建的功能分支上无缝执行命令
**以便** 不被不必要的提示打断工作流

**验收标准：**

- [ ] 当我在功能分支上执行命令时，不触发分支检测提示
- [ ] 命令正常执行，流程不受影响

### 故事 3：非 Git 环境中的使用

**作为** 开发者
**我想要** 在非 Git 仓库中使用 CodexSpec 命令
**以便** 在任何项目中都能使用需求澄清功能

**验收标准：**

- [ ] 在非 Git 仓库中执行命令时，跳过分支检测
- [ ] 命令正常执行，不报错

## 功能需求

### REQ-001：分支状态检测

系统必须能够检测当前 Git 分支状态：

- 获取当前分支名称
- 判断是否为主分支（main、master 或配置的其他主分支名）
- 判断是否在 Git 仓库中

### REQ-002：交互式分支创建提示

当检测到用户在主分支时，系统应：

- 显示当前分支信息
- 询问用户是否创建新功能分支
- 提供选项：创建新分支 / 在当前分支继续 / 取消操作

### REQ-003：分支创建执行

当用户选择创建分支时，系统应：

- 询问功能名称（用于分支命名）
- 调用 `scripts/bash/create-new-feature.sh` 或等效操作
- 创建格式为 `{prefix}-{feature-name}` 的分支
- 自动切换到新分支

### REQ-004：工作流继续

分支处理完成后，系统应：

- 无缝继续执行原有命令流程
- 不丢失用户之前提供的任何输入或上下文

## 非功能需求

### NFR-001：性能

- 分支检测操作应在 1 秒内完成
- 不应显著增加命令的响应时间

### NFR-002：兼容性

- 支持 Git 2.0 及以上版本
- 兼容 Bash 和 Zsh 环境
- 在无 Git 环境中优雅降级

### NFR-003：可配置性

- 主分支名称应可配置（默认：main, master）
- 检测行为应可通过配置启用/禁用

## 验收标准（测试用例）

### TC-001：主分支检测提示

**前置条件**：当前在 main 分支
**操作**：执行 `/codexspec:specify "新功能描述"`
**预期结果**：系统提示"您当前在主分支 main，是否创建新功能分支？"

### TC-002：创建新分支流程

**前置条件**：当前在 main 分支，选择创建新分支
**操作**：输入功能名称 "user-auth"
**预期结果**：创建分支并切换，如 `2026-0327-1825ab-user-auth`

### TC-003：跳过分支创建

**前置条件**：当前在 main 分支
**操作**：选择"在当前分支继续"
**预期结果**：不创建分支，直接继续执行原有命令

### TC-004：功能分支无提示

**前置条件**：当前在 `2026-0321-1430k7-user-auth` 分支
**操作**：执行 `/codexspec:specify "新功能描述"`
**预期结果**：不触发分支检测提示，直接执行命令

### TC-005：非 Git 环境处理

**前置条件**：在非 Git 仓库目录中
**操作**：执行 `/codexspec:specify "新功能描述"`
**预期结果**：跳过分支检测，正常执行命令

### TC-006：取消操作

**前置条件**：当前在 main 分支
**操作**：选择"取消操作"
**预期结果**：终止当前命令执行，返回用户控制

## 边界情况

### 边界 1：分离头指针（Detached HEAD）

- **情况**：用户在 detached HEAD 状态
- **处理**：视为非主分支状态，不触发提示，但可显示警告

### 边界 2：未提交的更改

- **情况**：主分支上有未提交的更改
- **处理**：提示用户存在未提交更改，建议先处理后再创建分支

### 边界 3：远程分支同步

- **情况**：本地 main 与远程 main 不同步
- **处理**：仅检测本地分支状态，不考虑同步状态

### 边界 4：自定义主分支名

- **情况**：项目使用 `develop` 或其他名称作为主分支
- **处理**：读取 `.codexspec/config.yml` 中的配置，如无配置则使用默认值

## 排除范围

- 不涉及 CLI 命令（`codexspec init`、`codexspec check` 等）的修改
- 不修改 `create-new-feature.sh` 脚本的现有行为
- 不实现跨仓库的分支检测
- 不处理 Git 冲突或合并操作
- 不实现分支推送或远程同步

## 实现范围

### 目录结构说明

CodexSpec 项目有两个相关的命令目录：

| 目录 | 用途 | 修改策略 |
|------|------|----------|
| `templates/commands/` | **源模板目录** - `codexspec init` 从这里复制模板到用户项目 | ✅ **需要修改** |
| `.claude/commands/codexspec/` | **当前项目使用的命令** - 由 `codexspec init` 安装 | ❌ 不应直接修改 |

> **重要**：修改应该只在 `templates/commands/` 中进行。当用户运行 `codexspec init` 时，模板会被复制到项目的 `.claude/commands/codexspec/` 目录。

### 需要修改的文件

| 文件 | 修改内容 |
|------|----------|
| `templates/commands/specify.md` | 添加 Git 分支安全检测指令 |
| `templates/commands/generate-spec.md` | 添加 Git 分支安全检测指令（可选） |

### 不需要修改的文件

- `src/codexspec/__init__.py` - CLI 入口无需修改
- `scripts/bash/create-new-feature.sh` - 脚本功能已完善
- `scripts/bash/common.sh` - 工具函数无需修改

## 技术约束

- 使用 Claude Code 的 Bash 工具执行 Git 命令
- 使用 AskUserQuestion 工具进行交互式询问
- 依赖现有的 `create-new-feature.sh` 脚本进行分支创建
