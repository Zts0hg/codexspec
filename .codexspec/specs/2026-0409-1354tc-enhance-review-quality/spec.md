# Feature: CodexSpec Review 评分机制与生成质量优化

## Overview

本功能旨在改进 CodexSpec 项目中 review 命令的评分机制和生成模板的质量标准，解决用户反馈的以下核心问题：

1. **评分不透明**：review 报告给出的分数缺乏清晰的计算依据，用户无法理解为什么得到某个分数
2. **评分不合理**：即使修复了所有 Critical Issues 和 Warnings，分数仍难以达到 100 分
3. **LLM 过于顺从**：当用户对分数提出疑问时，LLM 直接无条件修改分数，而不是重新评估验证
4. **生成质量不稳定**：生成的 spec/plan/tasks 经常存在问题，需要多次迭代才能通过 review

## Goals

- **评分准确性**：建立明确的评分细则（Rubrics），确保分数准确反映文档质量
- **评分可追溯性**：每个分数都有明确的计算依据，用户可以追溯扣分点
- **评分客观性**：建立评分验证机制，当用户质疑分数时，LLM 需要重新评估而非盲目修改
- **生成质量提升**：在生成模板中预定义质量目标，减少后续迭代次数

## User Stories

### Story 1: 评分依据透明化

**作为** CodexSpec 用户
**我希望** review 报告能清楚地展示每个分数的计算过程
**以便于** 我理解文档的不足之处并有针对性地改进

**Acceptance Criteria:**

- [ ] review 报告包含每个评分类别的评分标准对照
- [ ] review 报告列出每个扣分点及扣分理由
- [ ] 用户可以根据评分依据理解为什么得到某个分数
- [ ] 评分依据的格式统一且易于阅读

### Story 2: 合理的满分路径

**作为** CodexSpec 用户
**我希望** 只要修复了所有必须修复和应该修复的问题，就能达到接近满分的成绩
**以便于** 我的努力能够得到公正的反馈

**Acceptance Criteria:**

- [ ] "Suggestions (Nice to Have)" 类别对分数的影响有明确上限（最多 5 分）
- [ ] 修复所有 Critical Issues 和 Warnings 后，分数应 ≥ 95 分
- [ ] review 报告中明确区分"必须修复"、"应该修复"和"可选改进"的影响
- [ ] 评分规则在模板中有明确定义

### Story 3: 评分验证与质疑响应

**作为** CodexSpec 用户
**我希望** 当我对分数提出疑问时，LLM 能提供充分的依据并重新评估
**以便于** 评分保持客观性，不会因为我的质疑而盲目改变

**Acceptance Criteria:**

- [ ] 当用户质疑分数时，LLM 首先提供完整的评分依据
- [ ] LLM 询问用户具体哪个评分项不合理
- [ ] LLM 针对用户指出的具体项重新评估
- [ ] 如果重新评估后认为原分数合理，LLM 解释原因并维持原分数
- [ ] 如果重新评估后发现原分数确实不合理，LLM 调整分数并说明调整理由

### Story 4: 生成质量前置控制

**作为** CodexSpec 用户
**我希望** 生成的文档首次就能达到较高的质量标准
**以便于** 减少 review 后的迭代次数

**Acceptance Criteria:**

- [ ] 生成模板中包含明确的"质量目标"（Quality Targets）清单
- [ ] 质量目标与 review 评分标准对齐
- [ ] LLM 在生成过程中自检每个部分是否符合质量目标
- [ ] 生成模板添加"优秀示例"参考（可选）

## Functional Requirements

- [REQ-001] **评分细则（Rubrics）**：每个 review 模板必须包含明确的评分标准
  - 定义每个类别满分的标准
  - 定义扣分规则和具体扣分值
  - 提供评分示例

- [REQ-002] **建议项分数上限**：修改所有 review 模板的评分逻辑
  - Suggestions (Nice to Have) 类别最多影响 5 分
  - 或将建议项与评分完全分离显示

- [REQ-003] **评分依据展示**：review 报告必须包含完整的评分依据
  - 每个类别展示评分标准
  - 每个类别展示扣分明细（如有）
  - 确保用户可追溯每个分数的来源

- [REQ-004] **评分验证机制**：LLM 给出分数前必须完成验证清单
  - 验证每个扣分点都有明确依据
  - 验证分数计算正确
  - 验证分数与问题列表一致

- [REQ-005] **质疑响应流程**：定义用户质疑分数时的标准响应流程
  - 第一步：提供完整评分依据
  - 第二步：询问具体质疑项
  - 第三步：针对性重新评估并给出结论

- [REQ-006] **生成质量目标**：在所有生成模板中添加质量目标清单
  - spec-to-plan.md
  - plan-to-tasks.md
  - 其他生成模板

- [REQ-007] **生成与评分对齐**：确保生成时的质量目标与 review 评分标准一致
  - 使用相同的术语和定义
  - 使用相同的分类结构

- [REQ-008] **模板修改规范**：遵守宪法定义的目录结构
  - 仅修改 `templates/commands/` 下的源模板
  - 不修改 `.claude/commands/codexspec/` 下的活跃命令

## Non-Functional Requirements

- [NFR-001] **向后兼容**：改进后的模板应与现有工作流兼容
- [NFR-002] **语言支持**：评分细则和依据应支持项目配置的输出语言
- [NFR-003] **可维护性**：评分细则应清晰明确，便于后续维护和调整
- [NFR-004] **性能**：改进不应显著增加 LLM 的响应时间

## Acceptance Criteria (Test Cases)

- [TC-001] **评分准确性测试**：给定一个高质量文档，review 应给出 ≥ 95 分
- [TC-002] **评分依据完整性测试**：review 报告必须包含所有类别的评分依据
- [TC-003] **建议项影响测试**：仅存在 Suggestions 的文档应获得 ≥ 95 分
- [TC-004] **质疑响应测试**：用户质疑分数时，LLM 不应直接修改，而应提供依据
- [TC-005] **生成质量测试**：使用改进后的生成模板，首次生成的质量应有所提升
- [TC-006] **模板一致性测试**：所有 review 模板应使用统一的评分格式

## Edge Cases

- **Edge Case 1: 无宪法项目**
  - **处理**：review 模板应检测 constitution 是否存在，如不存在则跳过相关评分项
  - **影响**：Constitution Alignment 类别的权重应动态调整

- **Edge Case 2: 空文档或不完整文档**
  - **处理**：应给出低分（如 ≤ 30 分）并提供明确的改进建议
  - **不应**：给出模糊的中等分数

- **Edge Case 3: 用户反复质疑分数**
  - **处理**：每次质疑都应重新评估，但保持客观标准
  - **不应**：因用户压力而不断调整分数

- **Edge Case 4: 跨语言项目**
  - **处理**：评分细则应使用配置的输出语言
  - **技术术语**：可保持英文

## Out of Scope

- **自动修复功能**：本功能不包含自动修复 review 发现问题的能力
- **CLI 命令变更**：不修改现有的 CLI 命令结构（init, check, version, config）
- **新命令开发**：不开发新的 slash 命令类型
- **UI 界面**：不涉及任何 UI 界面开发（仅命令行交互）
- **外部工具集成**：不涉及与外部代码质量工具的集成
