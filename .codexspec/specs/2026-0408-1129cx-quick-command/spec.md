# Feature: `/codexspec:quick` — 一站式快速实现命令

## 概述

为 CodexSpec 项目新增 `/codexspec:quick` slash 命令，提供从需求澄清到代码实现的一站式自动化流程。该命令面向**小型需求**，通过自动化串联现有 SDD 命令（generate-spec → review-spec → spec-to-plan → review-plan → plan-to-tasks → review-tasks → implement-tasks），最大程度减少用户干预。用户只需提供初始需求描述并回答关键澄清问题。

## 目标

- 将 SDD 全流程自动化为一个命令，降低小型需求的使用门槛
- 复用现有命令模板，保持架构一致性
- 保留完整产物记录（spec.md、plan.md、tasks.md），便于追溯
- 在遇到重大问题时能回退并询问用户

## 功能需求

### REQ-001: 命令入口

- 命令名称：`/codexspec:quick`
- 接受用户输入的需求描述作为参数
- 模板文件位置：`templates/commands/quick.md`
- 需包含标准的语言偏好检查（读取 `.codexspec/config.yml`）
- 需包含标准的 Git 分支安全检查

### REQ-002: 复杂度评估

- 在开始流程前，对用户需求进行复杂度评估
- 评估维度：预估涉及文件数量、跨模块依赖数、新增/修改范围
- 评估结果及参考阈值：
  - **小型**（适合 quick）：预估 ≤3 个文件变更，涉及单模块，无外部依赖引入
  - **中型**（可尝试但建议标准流程）：预估 4-8 个文件变更，或跨 2 个模块
  - **大型**（建议标准流程）：预估 >8 个文件变更，或跨 3+ 个模块，或需引入新外部依赖
- 以上阈值为参考基准，agent 可根据需求特性灵活调整判断
- 对于中型/大型需求，使用 `AskUserQuestion` 工具向用户展示评估结果并提供选择

### REQ-003: 精简澄清

- 仅针对关键模糊点提问，通常控制在 2-5 个问题
- 使用 `AskUserQuestion` 工具提供结构化选项，减少用户输入负担
- 如果需求描述已足够清晰，可跳过澄清直接进入生成阶段

### REQ-004: 自动化 SDD 流程

- 按顺序原样调用以下现有命令模板（通过 Skill 工具），共 **4 步**：
  1. `generate-spec` — 生成 spec.md（内含自动调用 review-spec）
  2. `spec-to-plan` — 生成 plan.md（内含自动调用 review-plan）
  3. `plan-to-tasks` — 生成 tasks.md（内含自动调用 review-tasks）
  4. `implement-tasks` — 执行代码实现
- 3 个 review 命令由对应的生成命令内部自动调用，quick 不单独调用
- **调用原则**：子命令原样调用，不传递特殊前缀或跳过指令，不修改任何现有命令模板
- **数据传递方式**：generate-spec 自行创建 spec 目录，后续命令通过 spec 目录路径传参。例如：
  - `Skill("codexspec:spec-to-plan", args=".codexspec/specs/{prefix}-{name}/spec.md")`
  - `Skill("codexspec:implement-tasks", args=".codexspec/specs/{prefix}-{name}/tasks.md")`
- 每个 review 阶段的小问题由 agent 自行修正，不中断用户

### REQ-005: 重大问题回退机制

- **小问题（自动修正）**：格式缺失、措辞不精确、遗漏可合理推断的细节、轻微的结构调整
- **重大问题（暂停询问用户）**：
  - 需求间存在逻辑矛盾或冲突
  - 范围比初始描述扩大超过 50%
  - 技术上不可行的需求
  - 缺少无法推断的关键信息，必须由用户提供
- 判断原则：如果 agent 能基于上下文和最佳实践自信地做出修正，则为小问题；如果修正需要用户决策或涉及需求方向变更，则为重大问题
- 向用户清晰描述问题，获取反馈后从问题点继续

### REQ-006: 产物管理

- 由 generate-spec 命令在 `.codexspec/specs/` 下创建完整的 spec 目录（使用标准的时间戳+随机后缀命名方案）
- 目录中包含：spec.md、plan.md、tasks.md
- Review 报告也保存在同一目录下
- 流程完成后不自动执行 git 提交

### REQ-007: 流程状态反馈

- 在每个主要阶段（澄清、[1/4] spec 生成+审查、[2/4] plan 生成+审查、[3/4] tasks 生成+审查、[4/4] 实现）之间向用户显示进度信息
- 流程完成后提供最终总结，包括：创建的文件列表、实现概述、后续建议

## 非功能需求

### NFR-001: 模板一致性

- quick.md 模板须包含所有标准段落：YAML frontmatter、Language Preference 段落、Configuration Check 段落、Git Branch Safety Check 段落
- 段落结构和格式与现有命令模板保持一致（可参照 `generate-spec.md` 作为基准）

### NFR-002: 可维护性

- 通过 Skill 工具原样调用现有命令模板，不修改任何现有命令，确保 quick 命令随底层命令的更新自动受益
- quick.md 自身仅包含复杂度评估、精简澄清和流程编排逻辑，生成和审查逻辑完全由子命令承担

### NFR-003: 用户体验

- 自动流程中的澄清问题应使用 `AskUserQuestion` 工具的结构化选项格式
- 在非中断运行时，应提供清晰的阶段进度提示（如 `[1/7] 生成 spec.md ...`）

- 输入：清晰的小型需求描述

## 边缘情况

| 场景 | 处理方式 |
|------|----------|
| 用户输入为空或过于简短 | 提示用户补充需求描述 |
| 调用的子命令执行失败 | 向用户报告失败原因，提供手动继续的指引 |
| 用户在澄清中大幅扩展需求范围 | 重新评估复杂度，必要时建议切换至标准流程 |
| 项目无 `.codexspec/` 目录 | 提示用户先运行 `codexspec init` |
| config.yml 不存在 | 使用默认语言设置（en），显示一次性配置提示 |
| 上下文窗口接近极限 | 在调用子命令时精简中间输出，必要时省略 review 报告的详细内容，仅保留关键发现和修正摘要 |

## 澄清记录

### Session 2026-04-08

**Q1**: Git Branch Safety Check 是否在 7 个子命令中都存在？
**A1**: 不是。只有 specify 和 generate-spec 包含 Git Branch Safety Check，其他子命令没有。
**影响**: REQ-004（修正了对子命令内置行为的描述）

**Q2**: 是否需要「quick 模式」跳过机制来避免 Language Preference 重复检查？
**A2**: 不需要。Language Preference 检查是静默读取配置文件，幂等且轻量，重复执行无害。不引入特殊模式，子命令原样调用。
**影响**: REQ-004（简化了 Skill 调用设计，移除跳过机制）

**Q3**: 子命令是否应该原样调用，而非内联生成 spec？
**A3**: 是。子命令必须原样调用，不采用内联方式。generate-spec、spec-to-plan、plan-to-tasks 各自内含 auto-review 步骤，quick 实际只需调用 4 个命令（而非 7 个）。
**影响**: REQ-004（从 7 步简化为 4 步）、NFR-002（更新可维护性描述）

## 范围排除

- **不替代标准 SDD 流程** — quick 命令是标准流程的补充，适用于小型需求
- **不自动创建 Git 分支** — 但保留标准的分支安全检查提示
- **不自动 Git 提交** — 实现完成后由用户决定
- **不支持增量/部分执行** — 每次运行从头执行完整流程（如需部分执行应使用单独的命令）
- **不自动处理 CLI 注册** — 命令模板本身不需要 CLI 代码变更，仅作为 slash command 存在
